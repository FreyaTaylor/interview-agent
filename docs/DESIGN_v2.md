# 面试备考 Agent 系统 — 产品设计文档 v2.0

> 基于 v1.0 迭代，采纳了 MVP 拆分、评分 Rubric、自由探索边界控制、面经数据源等建议。
> 目标：在全新上下文中，任何人读完都能理解"我要做什么、为什么这么做"。

---

## 一、背景与意图

### 我是谁
- Java 后端开发者，转型 AI Agent 方向
- 需要**两件事同时达成**：高效准备面试（学习工具）+ 有一个能展示 Agent 技术深度的项目（面试项目）

### 核心痛点
- 知识体系零散，不知道先学什么、什么高频考
- 学了就忘，没有系统复习机制
- 面试完不知道哪里答得不好
- 面试 Agent 岗位，需要一个**真正用到 Agent 核心技术**的项目

### 一句话定位
**一个用 Agent 技术构建的个性化面试备考系统——既是我的学习工具，也是我的面试项目。**

---

## 二、核心抽象：一切都是"知识点 + 对话"

在设计功能之前，先确立一个统一抽象。不管数据来源是什么，最终都归到同一个结构：

```
知识点（知识树的叶子节点，最小学习单元）
  ├── 对应 N 个高频问题（每个问题附带评分 Rubric）
  ├── 我的每次回答 = 一次"对话"
  │     → 基于 Rubric 打分 + 纠错 + 标准答案
  │     → 可自由探索（追问/讨论，有边界控制）
  │     → 更新掌握度 + 存储理解向量
  └── 掌握度追踪（跨会话持久化）
```

### 不同来源，统一处理

| 来源 | 示例 | 处理方式 |
|------|------|---------|
| 面试录音/文本里的提问 | "分布式锁怎么实现？" | → 归到知识点 → 对话 |
| 学习模式出的题 | "Redisson 看门狗原理？" | → 归到知识点 → 对话 |
| 录音/文本里的项目拷打 | "你的订单超时取消怎么做？" | → 归到知识点 → 对话 |
| 录音/文本里的算法题 | "手撕 LRU" | → 特殊处理：匹配 LeetCode 题号，不进知识树 |
| 录音/文本里的 HR 题 | "为什么离职？" | → 简单记录，不纳入掌握度体系 |

**关键：不管是录音解析、文本上传还是学习模式出题，最终都进入完全一样的对话流程。**

### 知识点的粒度

```
分布式（一级）
  └── 分布式锁（二级）
        ├── Redis 分布式锁（三级 = 知识点 = 叶子节点）
        │     → 高频问题：SETNX+EX、看门狗续期、Lua 防误删、RedLock、主从丢锁
        ├── ZooKeeper 分布式锁（知识点）
        └── MySQL 分布式锁（知识点）
```

- **叶子节点 = 知识点 = 最小学习单元**
- "看门狗"不是知识点，是"Redis 分布式锁"这个知识点下的一个**高频问题**
- 一个知识点对应 N 个高频问题，掌握度在知识点级别追踪

### 录音/文本里 20 个提问怎么归类？

```
20 个提问 → LLM 聚类（同知识点的问题+追问归为一组）→ 9 个知识点

示例：
  Q1: "分布式锁怎么实现的？"        ┐
  Q2: "Redis SETNX 有什么问题？"     │→ 归为：Redis 分布式锁（1 个知识点）
  Q3: "看门狗怎么续期？"             │
  Q4: "主从切换锁会怎样？"           ┘

  Q5: "TCP 三次握手讲一下"           ┐
  Q6: "为什么不是两次？"             │→ 归为：TCP 三次握手（1 个知识点）
                                     ┘
  Q18: "手撕 LRU"                    → 算法题：匹配 LeetCode #146
  Q20: "你最大的优点缺点？"          → HR 题：记录，不进知识树
```

### 模块一：知识树

**做什么：** Agent 自主生成知识点树，每个叶子节点关联 N 个高频问题。**用户可编辑。**

**Agent 行为：**
- 先生成一级骨架（Java基础、MySQL、Redis...）
- 逐块展开到叶子节点
- 每个叶子节点标注面试权重（★1-5）
- 最后审查修剪（去重、补遗漏）
- LLM 自主决定展开顺序和深度

**用户编辑能力：**
- 用户可手动增删改知识树节点（如添加 "系统设计/秒杀系统"）
- 用户可调整节点权重
- 用户可合并/拆分节点
- Agent 生成的是**初始版本**，用户拥有最终决定权

**权重来源：**
- 初始：LLM 根据行业经验给出
- 持续更新：后台面经 pipeline 统计真实面试题频次 → 映射到知识点 → 更新权重
- 用户手动调整覆盖以上权重

### 模块二：学习（核心模块）

**两种输入方式，一个对话流程。**

#### 输入方式 A：上传面试录音/文本

**支持两种格式：**
- **文本上传（优先支持）**：用户直接粘贴/上传面试记录文本，开发成本最低，跳过 ASR 环节
- **录音上传（后续扩展）**：录音 → 语音转文字 → 等同于文本处理

> 💡 **设计决策**：先做文本上传，录音只是多一步 ASR，后续加即可。两者后续流程完全一致。

```
面试文本（或录音转写文本）
  → LLM 拆分问答对（20 个提问）
  → LLM 聚类为 N 个知识点（追问归到同一组）
  → 分类：
      知识点类 → 匹配知识树节点 → 进入"对话"流程
      算法题   → 匹配 LeetCode 题号 → 记录
      HR 题    → 简单记录
  → 汇总报告：涉及 N 个知识点，平均分，薄弱点排名
```

#### 输入方式 B：选择知识点学习

```
用户选择一个知识节点
  → Agent 检索用户历史掌握度 + 历史理解
  → 根据掌握度出题（不会→基础题，会了→进阶题）
  → 进入"对话"流程
```

#### 对话流程（统一，不管来源是什么）

```
一个知识点 + 我的回答（或 Agent 出题后我的回答）
  │
  ├── 基于 Rubric 打分 + 纠错 + 标准答案（详见"评分机制"）
  │
  ├── 用户可选择：
  │     ├── 自由探索（有边界控制，详见"探索边界"）
  │     │     → Agent 用 RAG 检索知识库+面经库回答
  │     │     → 多轮对话，短期 Memory 保持上下文
  │     │
  │     └── 直接下一题
  │
  ├── 更新掌握度
  ├── 存储本次理解（向量化，下次出题用）
  └── 推荐下一个知识点（遗忘曲线 + 权重 + 掌握度）
```

#### 评分机制：Rubric 评分（v2.0 新增）

**问题**：让 LLM 直接打 0-100 分，同一个答案可能一次 70 一次 85，掌握度追踪无意义。

**方案**：每个高频问题预设评分 Rubric，LLM 按 Rubric 评分。

```
示例：知识点 = Redis 分布式锁
问题 = "Redis 分布式锁怎么实现？"

Rubric（由 Agent 在生成知识树/问题时生成）：
  ┌──────────────────────────────┬──────┐
  │ 关键点                       │ 分值 │
  ├──────────────────────────────┼──────┤
  │ SETNX + EX 原子设置          │ 20   │
  │ value 用唯一标识防误删        │ 20   │
  │ Lua 脚本保证删除原子性        │ 20   │
  │ 看门狗续期机制               │ 20   │
  │ 主从切换丢锁 / RedLock       │ 20   │
  └──────────────────────────────┴──────┘

  → 用户提到了 3 个关键点 = 60 分
  → 可重复、可解释、可追踪
```

**Rubric 的生成与维护：**
- Agent 生成知识点时同步生成每个问题的 Rubric
- 面经 pipeline 采集到新的考察角度时，Agent 可自动补充 Rubric 条目
- 用户可手动编辑 Rubric（高级功能，非 MVP）

**项目经验类问题的评分：**

项目拷打题没有标准答案，但可以评估**回答质量维度**：

```
示例：知识点 = 项目经验/订单系统/超时取消方案
问题 = "订单超时取消怎么做的？"

Rubric（质量维度，非标准答案）：
  ┌──────────────────────────────┬──────┐
  │ 维度                         │ 分值 │
  ├──────────────────────────────┼──────┤
  │ 方案描述清晰完整              │ 25   │
  │ 解释为什么选这个方案（对比）   │ 25   │
  │ 踩过的坑 / 异常处理           │ 25   │
  │ 性能/扩展性考量               │ 25   │
  └──────────────────────────────┴──────┘

  → LLM 基于"这样的背景 + 这样的设计"评估表达质量
  → 不判断方案对错，判断讲述是否有深度、有逻辑
```

#### 自由探索的边界控制（v2.0 新增）

**问题**：用户追问可能完全跑题，导致上下文膨胀、掌握度评估失准。

**规则：**

| 规则 | 说明 |
|------|------|
| **相关性检查** | Agent 判断追问是否与当前知识点相关。不相关则温和拒绝并引导回正题 |
| **轮数上限** | 单个知识点的自由探索最多 5 轮（可配置），防止上下文过长 |
| **不影响评分** | 自由探索的内容**不影响**该知识点的掌握度评分。评分只看用户对问题的正式回答 |
| **可触发新知识点** | 如果追问涉及另一个知识点（如从 HashMap 追问到 ConcurrentHashMap），Agent 建议跳转到该知识点继续学习 |

```
用户："为什么负载因子是 0.75？"          → 相关，继续回答
用户："那 Go 的 map 是怎么实现的？"      → 偏离，提示：这属于另一个知识点，要跳转吗？
用户："今天吃什么？"                     → 不相关，温和拒绝
第 6 轮追问                              → 提示：探索已达上限，建议进入下一个知识点
```

#### 特殊处理

**算法题：**
- 不进入知识树掌握度体系
- 记录题目名称 → 匹配 LeetCode 题号/链接
- 用于后续刷题参考

**HR/行为题：**
- 简单记录题目和回答
- 不纳入掌握度计算

**项目拷打题：**
- 从面试文本/录音里提取
- 当作普通知识点处理：项目经验也是知识树的叶子节点
- 用质量维度 Rubric 评分（方案清晰度、对比分析、踩坑、扩展性）
- 示例：知识树里有 "项目经验/订单系统/超时取消方案"

### 推荐算法（学习流程内置）

```
priority = interview_weight × (1.0 - mastery_level) × decay(days_since_last_study)

高频考点 + 不会 → priority 最高 → 优先推荐
已掌握 → priority 低 → 不推荐（除非遗忘曲线触发复习）
```

### 遗忘曲线参数设计

采用简化版艾宾浩斯模型：

```
retention = e^(-t / S)

t = 距离上次学习的天数
S = 稳定性参数（越高记得越久）
```

**S 的确定与更新规则：**

| 场景 | S 值 | 说明 |
|------|------|------|
| 首次学习，得分 ≥ 80 | S = 3 | 学得好，3 天后复习 |
| 首次学习，得分 < 80 | S = 1 | 学得差，1 天后复习 |
| 复习时得分 ≥ 80 | S = S × 2 | 记住了，间隔翻倍（类 Anki） |
| 复习时得分 < 80 | S = max(1, S × 0.5) | 忘了，间隔减半 |

**触发复习的阈值：** 当 `retention < 0.5` 时将该知识点重新加入推荐队列。

**简单直觉：**
```
第 1 次学好了 → 3 天后复习
第 2 次还记得 → 6 天后复习
第 3 次还记得 → 12 天后复习
某次忘了     → 间隔减半，回退
```

> 💡 这套参数不需要精确。核心目的是让"快忘的"优先出现，而不是精确预测记忆。后续可根据实际数据调参。

## 四、为什么用 Agent？（Why Agent）

### 核心判断：有没有"决策不确定性"

| 场景 | 写死代码能做吗？ | 为什么需要 Agent |
|------|-----------------|-----------------|
| 生成知识树 | 一次 LLM 调用会被截断 | Agent 自己规划多步策略 |
| 出什么题 | 随机出题不管你会不会 | 根据掌握度+历史理解动态决定 |
| 追不追问 | 写不完的 if/else | LLM 看完回答自己判断要不要追问 |
| 什么时候停 | 固定 N 题 | LLM 判断"掌握了"→停止 |
| 推荐什么 | 固定排序 | 综合掌握度+遗忘+紧迫度判断 |
| 文本归类 | 规则匹配不准 | LLM 理解语义做聚类 |

**面经采集 pipeline 没有不确定性（固定步骤）→ 不用 Agent。**
**学习过程充满不确定性 → 必须用 Agent。**

### 每个功能展示的 Agent 概念

| Agent 概念 | 具体体现 |
|-----------|---------|
| **Planning** | 知识树生成：LLM 自己规划"骨架→展开→修剪"的多步策略 |
| **ReAct** | 学习对话：出题→观察回答→决定追问还是结束→循环 |
| **Tool Calling** | Agent 自主调用：查知识树、Rubric 评分、更新掌握度、检索面经、匹配 LeetCode... |
| **Memory（短期）** | 一次对话内的上下文保持 |
| **Memory（长期）** | 用户对每个知识点的理解，向量化存储，跨会话持久化 |
| **RAG** | 出题时从知识库+面经题库+用户历史理解中检索，个性化出题和回答 |

### Agent 的价值不是"能做什么"，而是"不需要预判所有情况"

```python
# 传统方式：穷举
if score < 60:
    ask_followup()
elif "关键词A" not in answer:
    ask_about_A()
# ... 写不完

# Agent 方式：LLM 自己判断
# "用户回答了扩容倍数但漏了 rehash 过程，这是常见坑，我应该追问"
```

## 五、同类产品对比

| 项目 | 做了什么 | 我的差异化 |
|------|---------|----------|
| Offer-Patato | 输入面经→生成答案 | 只有答案，没有知识树/掌握度/遗忘曲线 |
| Dynamic-Review-Agent | 遗忘曲线复习 | 权重是猜的，不是面经数据驱动 |
| vidyamind | 知识图谱+掌握度 | 没有面经数据源，没有录音复盘 |
| Mock-Interview-Agent | RAG+模拟面试 | 只出题，没有掌握度追踪 |
| 各种 mock interview | 模拟面试 | 通用题，不知道你哪里薄弱 |

**市面上都是单点工具。我做的是闭环系统：**

```
面经采集 → 知识树+权重 → 个性化出题 → Rubric 评分 → 掌握度 → 遗忘曲线推荐 → 再出题
     ↑                                                                    │
     └───────────────── 数据持续更新 ────────────────────────────────────────┘
```

没有任何一个项目同时做到：**真实面经数据 + 知识树 + 个性化对话 + Rubric 评分 + 掌握度追踪 + 遗忘曲线 + 面试复盘**。

## 六、技术选型宏观决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 语言 | Python | Agent 岗位默认语言；LangGraph 只有 Python 成熟；现有 pipeline 复用 |
| Agent 框架 | LangGraph | 支持状态图+条件边+循环，匹配 ReAct 模式 |
| LLM | DeepSeek + DashScope | 文本推理 + 图片 OCR + 语音转文字 + Embedding |
| 存储 | PostgreSQL + pgvector | 结构化数据 + 向量搜索，一个库搞定 |

### 降低 LLM 调用成本的策略

虽然 DeepSeek 已经很便宜，但一次学习会话（出题 + 打分 + 追问 + RAG + 更新）涉及 5-10 次 LLM 调用，长期使用仍需注意成本：

| 策略 | 说明 |
|------|------|
| **小模型分流** | 分类/判断类任务（"这个追问是否相关？"）用小模型或低价模型（如 DeepSeek-Chat），推理类用强模型 |
| **缓存高频问题** | 同一个高频问题的标准答案 + Rubric 只需生成一次，缓存后复用 |
| **批量处理** | 面试文本的聚类、分类可以一次 LLM 调用处理多条，而不是逐条调用 |
| **Prompt 精简** | 仅注入必要上下文，避免每次把整棵知识树塞进 prompt |
| **向量检索先行** | RAG 先用 pgvector 检索缩小范围，再把少量结果喂给 LLM，减少 token |

## 七、面经采集 Pipeline（v2.0 明确数据源）

### 数据源：小红书面经帖

**为什么选小红书：**
- 大量真实面经分享（"xx 公司 Java 后端面经"）
- 内容粒度适中（一篇帖子 = 一次面试的所有题目）
- 更新频繁，能反映当前面试趋势

**Pipeline 流程：**

```
小红书面经帖
  → 爬取（关键词：Java 面经、后端面经、xx 公司面试...）
  → 清洗（去广告、去重、提取纯面试题列表）
  → LLM 解析：每道题 → 映射到知识树节点
  → 统计：每个知识点被考到的频次
  → 更新知识树权重

定时运行（如每周），持续反映面试趋势变化
```

**合规注意：**
- 仅采集公开帖子，不涉及用户隐私
- 数据仅用于统计面试题频次，不存储/展示原文
- 爬取频率合理，不对平台造成压力

## 八、开发路线（v2.0 拆分更细）

```
Phase 0 — 对话流程验证（最小闭环）
  ┌─────────────────────────────────────────────────┐
  │ 目标：跑通 LangGraph 的 ReAct 循环               │
  │                                                 │
  │ - 硬编码几个知识点 + 高频问题 + Rubric            │
  │ - 学习对话流程：出题 → 用户回答 → Rubric 打分     │
  │ - 自由探索（带边界控制）                          │
  │ - CLI 交互即可，不做前端                          │
  │                                                 │
  │ 交付物：一个能在终端对话的学习 Agent               │
  └─────────────────────────────────────────────────┘

Phase 1 — 知识树 + 掌握度持久化
  ┌─────────────────────────────────────────────────┐
  │ - 知识树生成 Agent（Planning）                    │
  │ - 用户可编辑知识树                                │
  │ - Rubric 自动生成（随知识点生成）                  │
  │ - 掌握度存储 + 遗忘曲线推荐                       │
  │ - PostgreSQL + pgvector 持久化                    │
  │ - 用户历史理解向量化存储                          │
  └─────────────────────────────────────────────────┘

Phase 2 — 面试复盘 + 面经 Pipeline
  ┌─────────────────────────────────────────────────┐
  │ - 面试文本上传 → 聚类知识点 → 复用对话流程         │
  │ - 算法题匹配 LeetCode                            │
  │ - 汇总报告（涉及知识点、平均分、薄弱点）          │
  │ - 小红书面经 pipeline → 权重更新                  │
  │ - （可选）录音上传 → ASR → 转文本 → 复用流程      │
  └─────────────────────────────────────────────────┘

Phase 3 — 前端 + 体验
  ┌─────────────────────────────────────────────────┐
  │ - 前端技术栈待讨论（一人开发，优先低成本方案）     │
  │ - 知识树可视化（思维导图 / 树形编辑器）            │
  │ - 学习数据看板（掌握度热力图、学习曲线）           │
  │ - 对话界面                                       │
  └─────────────────────────────────────────────────┘
```

### Phase 0 的具体边界

Phase 0 只做最核心的事，验证两个假设：
1. LangGraph ReAct 循环能支撑学习对话的体验
2. Rubric 打分比自由打分稳定

**Phase 0 包含：**
- 3-5 个硬编码知识点 + 对应高频问题 + 手写 Rubric
- LangGraph 状态图：出题 → 等待回答 → Rubric 打分 → 自由探索（5 轮上限）→ 下一题
- CLI 交互

**Phase 0 不包含：**
- 知识树生成
- 数据库
- 掌握度持久化
- 面试文本解析
- 前端

## 十、面试怎么讲

### 一句话
"我做了一个面试备考 Agent——用 LangGraph 实现知识树生成（Planning）、个性化出题和追问（ReAct + Memory + RAG），Rubric 评分保证可追踪，后台面经 pipeline 从小红书采集真实数据驱动知识点权重。"

### 关键追问

| 问题 | 要点 |
|------|------|
| 为什么用 Agent？ | 学习有决策不确定性：出什么题、追不追问、何时停——需要 LLM 自主决策 |
| 为什么选 LangGraph？ | 状态图+条件边+循环，匹配 ReAct；AgentExecutor 不支持细粒度状态控制 |
| Memory 怎么做？ | 短期=State；长期=pgvector 存用户理解向量，出题时 RAG 检索 |
| 权重怎么来？ | LLM 初始化 + 小红书面经 pipeline 统计真实频次更新 |
| 打分怎么保证一致？ | 每个问题预设 Rubric（3-5 个关键点），LLM 按 Rubric 逐项判定，非自由打分 |
| 追问怎么控制？ | 相关性检查 + 5 轮上限 + 不影响评分 + 可触发知识点跳转 |
| 和 ChatGPT 直接问有什么区别？ | ChatGPT 没记忆、没知识树、不知道你薄弱点、不追问、没遗忘曲线 |
| 成本怎么控制？ | 小模型分流 + 缓存 Rubric + 批量处理 + Prompt 精简 + 向量检索先行 |

| 变更 | 说明 |
